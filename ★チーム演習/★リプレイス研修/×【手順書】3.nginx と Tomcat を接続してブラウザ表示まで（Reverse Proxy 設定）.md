# nginx と Tomcat を接続してブラウザ表示まで（Reverse Proxy 設定）

---

## 概要

本手順では Webサーバ（nginx）から APサーバ（Tomcat）へリバースプロキシし、  
ブラウザで「WebサーバのIP」にアクセスすると Tomcat の画面が表示される状態を作る。

実施したこと

- Tomcat が AP 側で 8080 待ち受けしていることを確認
- nginx 側に upstream + proxy_pass 設定を追加
- nginx 設定チェック → reload
- ブラウザ表示OKまで確認

---

## 構成イメージ
```bash
Browser
↓  http://<WEB_IP>/
nginx（Webサーバ :80）
↓  proxy_pass
Tomcat（APサーバ :8080）
```
---

# 1 Tomcat 側の疎通確認（nginx設定前の事前確認）

## 操作サーバ
Webサーバ（nginx側）または疎通可能なサーバ

## この工程でしていること
- 「APサーバのTomcatが動いているか」を先に確認する
- nginx設定に入る前に、アプリ側が生きていることを確定させる

## 実行コマンド
```bash
curl -i http://<AP_PRIVATE_IP>:8080/
```

## コマンドの意味

### curl
HTTPリクエストを送ってレスポンスを確認するコマンド

### -i
レスポンスヘッダ（HTTP/1.1 200 など）も表示する

## 確認方法とOKの目安

OKの目安

- HTTP/1.1 200 が返る
- 本文に Tomcat のHTML（Apache Tomcat/9.0.12）が出る

意味

- 「AP:8080」へのHTTP到達ができている
- 以降は nginx がそのAPへ代理接続できるように設定するだけ

# 2 nginx のメイン設定確認（競合を避ける前提確認）

## 操作サーバ
Webサーバ（nginx側）

## この工程でしていること
- 既存の nginx.conf で server { listen 80; } が存在するか確認する
- conf.d の server と競合しないように整理する準備

## 実行コマンド
```bash
sudo vi /etc/nginx/nginx.conf
```

## コマンドの意味

- /etc/nginx/nginx.conf：nginxのメイン設定ファイル
- server定義がここに残っていると、conf.d側と競合しやすい

## 確認方法とOKの目安

確認観点

- http { } 内で include /etc/nginx/conf.d/*.conf; が有効になっている
- nginx.conf 直下に default server が残っていないことが理想

OKの目安

- 仮想ホスト定義は conf.d 側に寄せる方針になっている

# 3 tomcat 用の VirtualHost 設定を作成（conf.d）

## 操作サーバ
Webサーバ（nginx側）

## この工程でしていること
- nginxがAPへ転送するための設定ファイルを追加する
- upstream（接続先の束）と server（受け口）を作る

## 実行コマンド
```bash
vi /etc/nginx/conf.d/tomcat.conf
```

## 設定ファイルの内容

注意
- <AP_PRIVATE_IP> は APサーバのプライベートIPに置き換える

```bash
upstream tomcat_backend {
server <AP_PRIVATE_IP>:8080;
keepalive 32;
}

server {
listen 80;
server_name _;

location = /health {
    return 200 "ok\n";
    add_header Content-Type text/plain;
}

location / {
    proxy_pass http://tomcat_backend;

    proxy_http_version 1.1;
    proxy_set_header Connection "";

    proxy_set_header Host              $host;
    proxy_set_header X-Real-IP         $remote_addr;
    proxy_set_header X-Forwarded-For   $proxy_add_x_forwarded_for;
    proxy_set_header X-Forwarded-Proto $scheme;

    proxy_connect_timeout 5s;
    proxy_read_timeout    60s;
    proxy_send_timeout    60s;
}

}
```
---

## 設定の意味（重要）

### upstream tomcat_backend { ... }

この工程でしていること
- 転送先（Tomcat）を名前（tomcat_backend）で定義する

各行の意味

- server <AP_PRIVATE_IP>:8080;
  - proxy_pass の宛先（Tomcatの待ち受け）を指定する
  - 8080 は Tomcat のHTTPポート

- keepalive 32;
  - nginx と Tomcat の接続を使い回す（接続の再利用）
  - 性能改善（毎回TCP接続を作らない）

---

### server { listen 80; server_name _; }

この工程でしていること
- 「nginxが80で受ける」仮想ホストを作る

各行の意味

- listen 80;
  - HTTPの受け口を80にする（ブラウザが普通にアクセスできる）

- server_name _;
  - どのHostヘッダでも受ける（ワイルドカード的な扱い）
  - IP直打ちアクセスの受け口としても使いやすい

注意点
- server_name "_" は「同じ80番で同名のserver」があると競合する

---

### location = /health { ... }

この工程でしていること
- ヘルスチェック用の軽いエンドポイントを返す
- Tomcatを経由せず nginx 単体で生存確認できる

各行の意味

- return 200 "ok\n";
  - 200 OK と本文 ok を返す

- add_header Content-Type text/plain;
  - ヘルスチェックをテキストとして返す

---

### location / { proxy_pass ... }

この工程でしていること
- 受けたリクエストを Tomcat へ転送する（Reverse Proxy）

重要な行

- proxy_pass http://tomcat_backend;
  - upstream に定義した Tomcat 宛へ転送する
  - 「/」配下全部をTomcatに渡す

- proxy_http_version 1.1;
  - TomcatへHTTP/1.1で話す
  - keep-aliveやWebアプリの動作が安定しやすい

- proxy_set_header Host $host;
  - クライアントが指定したホスト名をTomcat側へ渡す
  - アプリがHostを参照する場合に必要

- proxy_set_header X-Real-IP $remote_addr;
  - 元クライアントIPをTomcatへ渡す

- proxy_set_header X-Forwarded-For $proxy_add_x_forwarded_for;
  - 経由したIP情報を連鎖させる（ログ・監査で重要）

- proxy_set_header X-Forwarded-Proto $scheme;
  - 元のスキーム（http/https）を渡す
  - HTTPS化したときにアプリ側の判定に使う

タイムアウト系

- proxy_connect_timeout 5s;
  - Tomcatへの接続確立待ち

- proxy_read_timeout 60s;
  - Tomcatのレスポンス待ち

- proxy_send_timeout 60s;
  - nginx→Tomcat送信待ち

# 4 nginx 設定チェック（反映前の必須工程）

## 操作サーバ
Webサーバ（nginx側）

## この工程でしていること
- 設定ファイルの文法エラーを事前に検出する
- 誤設定で nginx が落ちる事故を防ぐ

## 実行コマンド
```bash
sudo nginx -t
```

## コマンドの意味

- -t：設定ファイルの構文チェック（起動はしない）

## OKの目安

- syntax is ok
- test is successful

注意
- warning がある場合は設計上の競合を疑い、可能なら解消する
  - server_name を具体名にする
  - 既存 default_server を無効化する
  - listen 80 default_server の整理を行う

# 5 nginx 反映（reload）

## 操作サーバ
Webサーバ（nginx側）

## この工程でしていること
- nginx を停止せず設定を読み直して反映する

## 実行コマンド
```bash
sudo systemctl reload nginx
```

## コマンドの意味

- reload：プロセスを落とさず設定を再読み込み

## 確認方法とOKの目安
```bash
systemctl status nginx
```

OKの目安
- active (running)
